# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Hugo Extended static site for **Bergen Ship Agency**, a Norwegian port agency (Bergen-based). Production domain: `https://bergenshipagency.no/`. Content is in English. Customer-facing marketing site ‚Äî no backend, no JS framework, no package manager.

## Commands

- `hugo server -D` ‚Äî local dev server with drafts (default http://localhost:1313).
- `hugo` ‚Äî production build to `public/`.
- `hugo new content services/<slug>.md` ‚Äî scaffold from `archetypes/default.md` (creates a draft).

Hugo **Extended** is required (Hugo Pipes minifies CSS/JS at build time ‚Äî see `layouts/_default/baseof.html`). There is no Node toolchain, no test suite, no linter.

## Architecture

### Custom layouts, no theme

`themes/` is intentionally empty. All templates live under `layouts/` and override Hugo defaults directly. When adding pages, prefer extending the existing layout system over introducing a theme.

Layout lookup that matters here:

- `layouts/_default/baseof.html` ‚Äî single base wrapping every page. Loads Google Fonts, minified+fingerprinted `main.css`/`main.js` with SRI integrity hashes, and pulls in `head-meta` (SEO) and `structured-data` (JSON-LD) partials.
- `layouts/partials/head-meta.html` ‚Äî per-page `<title>`, description, canonical, Open Graph, Twitter card. OG image is generated by `.Fill "1200x630 webp q85"` on `assets/images/<og_image>`.
- `layouts/partials/structured-data.html` ‚Äî JSON-LD. Home + contact emit `LocalBusiness` (with `PostalAddress`, `GeoCoordinates`, 24/7 `OpeningHoursSpecification`). Service pages emit `Service` with the LocalBusiness nested as `provider`. All other pages emit nothing.
- `layouts/index.html` ‚Äî homepage (hero + services grid + why-us + CTA). The services grid is **data-driven**: ranges over `(site.GetPage "/services").Pages.ByWeight` and delegates each card to `partials/service-card.html`.
- `layouts/_default/list.html` ‚Äî `/services/` listing, also ranges `.Pages.ByWeight` through the same `service-card` partial.
- `layouts/_default/single.html` ‚Äî fallback for generic single pages (about-us uses this).
- `layouts/services/single.html` ‚Äî service detail with right-side sidebar (`Our Services` nav + 24/7 contact widget). Sidebar list is generated from `(site.GetPage "/services").Pages.ByWeight` with `.active` applied to the current page.
- `layouts/contact/single.html` ‚Äî contact page is routed here because `content/contact.md` sets `type: "contact"` in front matter.
- `layouts/partials/header.html`, `footer.html` ‚Äî global nav and footer. Services dropdown (header) and services links (footer) are generated from the services section.
- `layouts/partials/service-card.html` ‚Äî single source of truth for service-card markup (home grid and `/services/` listing both use it). Reads `.Params.image`, `.Params.icon`, `.Params.summary`, `.Title`, `.RelPermalink`.
- `layouts/partials/norway-map.html` ‚Äî stylized SVG of Norway with port pins. Port list (Troms√∏, Bod√∏, Trondheim, √Ölesund, Bergen, Stavanger, Kristiansand) is hardcoded in the partial as a slice of dicts. Bergen is rendered as the HQ pin (gold + pulse animation); others are dark navy dots. The Norway outline is **intentionally stylized** ‚Äî not GIS-accurate. To edit ports, modify the `$ports` slice at the top of the partial.
- `layouts/partials/service-process.html` ‚Äî generic 4-step "How a port call works" timeline rendered on every service detail page. Step copy comes from `i18n/*.yaml` keys `process_step_{1..4}_{title,body}`, so editing one place updates every service page in both languages.
- `layouts/shortcodes/asset-image.html` ‚Äî `{{</* asset-image src="..." alt="..." width="..." */>}}` shortcode for embedding `assets/images/` images inside markdown bodies with Hugo image processing. Errors if `src` doesn't resolve.

### Content model

- `content/_index.md` ‚Äî home (title only; body is unused, homepage is layout-driven).
- `content/about-us.md`, `content/contact.md` ‚Äî top-level pages.
- `content/services/_index.md` + `content/services/<slug>.md` ‚Äî service section.
- Markdown uses **unsafe goldmark** (`hugo.yaml`: `markup.goldmark.renderer.unsafe: true`) ‚Äî service pages embed raw `<div class="service-features-grid">` HTML inside markdown. This is intentional; don't strip it.

**Service front matter contract** (load-bearing for the data-driven nav/cards):

```yaml
title: "Yacht"
image: "yacht.jpg"        # filename under assets/images/services/
icon: "M3 13.5L12 3..."   # SVG path "d" attribute, rendered inside <svg viewBox="0 0 24 24">
description: "..."        # SEO meta description
summary: "..."            # marketing copy used on home/listing cards (overrides Hugo's auto .Summary)
weight: 1                 # sort order across nav, dropdown, footer, sidebar, cards
```

`image` is a filename, not a path ‚Äî the template prefixes `images/services/`. `icon` is just the path `d` data, not full SVG markup.

### Site params

`hugo.yaml` `params`:

- Flat: `phone`, `email`, `address`, `description`, `og_image` (filename under `assets/images/`).
- Nested `business:` ‚Äî `legalName`, `streetAddress`, `postalCode`, `locality`, `country` (ISO 3166-1 alpha-2), `areaServed`, `latitude`, `longitude`. Drives the JSON-LD LocalBusiness emitted from `structured-data.html`.

`disableKinds: [taxonomy, term]` is set because the site has no categories/tags and we don't want `/categories/` or `/tags/` polluting the sitemap.

### Styling

- `assets/css/main.css` ‚Äî single hand-written stylesheet, no framework. Brand tokens are CSS custom properties at `:root`: `--navy` `#0a1628`, `--gold` `#c4922a`, fonts `Playfair Display` (serif headings) + `Source Sans 3` (sans body).
- `assets/js/main.js` ‚Äî vanilla JS for navbar scroll state, mobile menu toggle (with `aria-expanded` sync), dropdown toggle.
- Both pass through `resources.Minify | resources.Fingerprint` in `baseof.html` and emit `integrity="sha256-..."` for SRI ‚Äî edit the `assets/` source, not `public/`.

### Accessibility

- `.skip-link` in `baseof.html` jumps focus to `#main` (the `<main>` element) ‚Äî visually hidden until focused.
- `nav-toggle` keeps `aria-expanded` in sync with the `.open` class on the menu (see `main.js`).
- Outstanding: services dropdown is mouse-hover-only on desktop ‚Äî no keyboard support yet.

### Images

- All site images live under `assets/images/` (services in `assets/images/services/`). Hugo Pipes transcodes them to webp + fingerprints the output (cache-busting via filename hash).
- Service templates and the home page card use `.Process "webp q85"` (format-only, no resize) so the four service images and one about-us image dedupe to one built artifact each.
- The `asset-image` shortcode is the only way to surface `assets/images/` content from inside a markdown body ‚Äî direct `<img src="/images/...">` skips the pipeline and breaks if filenames change.
- Source dimensions: services 600√ó537, about-us 960√ó540, hero-yacht 3488√ó2616. Templates **don't upscale**; for sharper retina detail on services/about, replace the source with a larger image.
- **Hero background** is set via inline `style="background-image: url(...)"` in `layouts/index.html` (line ~3) so the URL goes through Hugo Pipes and gets fingerprinted. The CSS class `.hero-bg` only sets `background-position` / `-size` / `-repeat`. Don't put a `url(...)` back into the CSS ‚Äî it would break the migration.

### Multilingual

Two languages configured in `hugo.yaml`: `en` (default, served at `/`) and `nb` (Norwegian Bokm√•l, served at `/nb/`). `defaultContentLanguageInSubdir: false` keeps English at the root.

- **UI strings** ‚Üí `i18n/en.yaml` and `i18n/nb.yaml`. Flat snake_case keys (`nav_home`, `form_email`, etc.). Templates reference via `{{ i18n "key" }}`. Page bodies and marketing copy do NOT belong here ‚Äî those live in `content/*.md`.
- **Content translations** ‚Üí `*.nb.md` siblings of `*.md` files. Hugo auto-links them as translations of the same logical page. A missing translation falls back to a 404 ‚Äî there's no automatic English fallback for missing pages.
- **Current state** ‚Äî Norwegian stubs exist for top-level pages (home, about-us, contact, services index) with placeholder bodies marked `<!-- TODO -->`. Service detail pages (`yacht.nb.md` etc.) are not yet created; `/nb/services/<slug>/` will 404 and the Norwegian services list/dropdown will be empty until you create them.
- **Language switcher** lives in `partials/header.html` topbar, ranges over `.AllTranslations`. Marks current with `class="active"` + `aria-current="true"`.
- **Per-language params** ‚Äî each language block sets its own `description` (for `<meta description>` and OG description) and `og_locale` (`en_US` / `nb_NO`, Facebook-format territory codes). Other params (`phone`, `email`, `address`, `business.*`) are shared at the top level.
- **Internal links** must go through `relLangURL` (e.g. `{{ "/about-us/" | relLangURL }}`) so they prefix `/nb/` correctly when rendered in Norwegian context.
- **Form JS strings** are passed from Hugo to JS via `data-msg-*` attributes on `#contact-form` so the JS reads the localized text rather than hardcoding English.

### Adding a new UI string

1. Add the key + English text to `i18n/en.yaml`.
2. Add the same key + Norwegian to `i18n/nb.yaml` (mandatory ‚Äî Hugo logs a warning at build if a key is missing in a language).
3. Reference as `{{ i18n "key" }}` in templates.

### Translating a content page

1. Copy `content/foo.md` ‚Üí `content/foo.nb.md`.
2. Translate the `title`, `description`, and body.
3. Leave `image`, `icon`, `weight` as-is (those are language-independent).

## Adding a new service

One file, one image. Nav/cards/dropdown/sidebar/footer all update automatically.

1. Drop the image into `assets/images/services/<slug>.jpg` (or .png).
2. Create `content/services/<slug>.md` with the front-matter contract above (`title`, `image`, `icon`, `description`, `summary`, `weight`).

The home grid CSS is sized for 4 cards ‚Äî adding a 5th may need a CSS tweak in `assets/css/main.css` `.services-grid`.

## Placeholders the client should replace

- **Hero stats** (`layouts/index.html`): "24/7", "Core Services", "Bergen" ‚Äî replace with real operational numbers when you have them.
- **Footer memberships** (`layouts/partials/footer.html`): three `<span class="cert-placeholder">` chips (ISO 9001, Norwegian Shipowners' Assoc., IMO ISM) are dashed-outline placeholders. Replace with real certification/membership logos (use the `asset-image` shortcode pattern).
- **LinkedIn URL** (`layouts/partials/footer.html`): currently points at `https://www.linkedin.com/` ‚Äî set to the real company page URL.
- **Exclusive Scandinavia** callout: hardcoded as plain text in `partials/footer.html`. If they're an actual partner, link to their site.
- **Norway coverage ports** (`layouts/partials/norway-map.html`): seeded with 7 major ports ‚Äî adjust the `$ports` slice to match actual coverage. Pin coordinates are tied to the SVG `viewBox="0 0 240 600"` ‚Äî recalibrate `x`/`y` if you change the outline path.

## Gotchas

- `static/images/` exists but is empty ‚Äî Hugo Pipes (`assets/`) is the canonical image location now.
- `.hugo_build.lock` is a Hugo runtime artifact; leave it alone.
- `public/` is build output (ignored by git); never edit it by hand.
- `resources/_gen/` is the image-processing cache (also ignored). Safe to delete; Hugo will regenerate.
- Local Hugo install: this machine's snap is broken (`/snap/hugo` missing `meta/snap.yaml`). Use `~/.local/bin/hugo` (Hugo Extended 0.151.0 installed manually). Repair the snap with `sudo snap refresh hugo` when convenient.
