## Project

Hugo Extended static site for **Bergen Ship Agency**, a Norwegian port agency (Bergen-based). Production domain: `https://bergenshipagency.no/`. Content is in English. Customer-facing marketing site â€” no backend, no JS framework, no package manager.

## Commands

- `hugo server -D` â€” local dev server with drafts (default http://localhost:1313).
- `hugo` â€” production build to `public/`.
- `hugo new content services/<slug>.md` â€” scaffold from `archetypes/default.md` (creates a draft).

Hugo **Extended** is required (Hugo Pipes minifies CSS/JS at build time â€” see `layouts/_default/baseof.html`). There is no Node toolchain, no test suite, no linter.

## Architecture

### Custom layouts, no theme

`themes/` is intentionally empty. All templates live under `layouts/` and override Hugo defaults directly. When adding pages, prefer extending the existing layout system over introducing a theme.

Layout lookup that matters here:

- `layouts/_default/baseof.html` â€” single base wrapping every page. Loads Google Fonts, minified+fingerprinted `main.css`/`main.js` with SRI integrity hashes, and pulls in `head-meta` (SEO) and `structured-data` (JSON-LD) partials.
- `layouts/partials/head-meta.html` â€” per-page `<title>`, description, canonical, Open Graph, Twitter card. OG image is generated by `.Fill "1200x630 webp q85"` on `assets/images/<og_image>`.
- `layouts/partials/structured-data.html` â€” JSON-LD. Home + contact emit `LocalBusiness` (with `PostalAddress`, `GeoCoordinates`, 24/7 `OpeningHoursSpecification`). Service pages emit `Service` with the LocalBusiness nested as `provider`. All other pages emit nothing.
- `layouts/index.html` â€” homepage (hero + services grid + why-us + CTA). The services grid is **data-driven**: ranges over `(site.GetPage "/services").Pages.ByWeight` and delegates each card to `partials/service-card.html`.
- `layouts/_default/list.html` â€” `/services/` listing, also ranges `.Pages.ByWeight` through the same `service-card` partial.
- `layouts/_default/single.html` â€” fallback for generic single pages (about-us uses this).
- `layouts/services/single.html` â€” service detail with right-side sidebar (`Our Services` nav + 24/7 contact widget). Sidebar list is generated from `(site.GetPage "/services").Pages.ByWeight` with `.active` applied to the current page.
- `layouts/contact/single.html` â€” contact page is routed here because `content/contact.md` sets `type: "contact"` in front matter.
- `layouts/partials/header.html`, `footer.html` â€” global nav and footer. Services dropdown (header) and services links (footer) are generated from the services section.
- `layouts/partials/service-card.html` â€” single source of truth for service-card markup (home grid and `/services/` listing both use it). Reads `.Params.image`, `.Params.icon`, `.Params.summary`, `.Title`, `.RelPermalink`.
- `layouts/shortcodes/asset-image.html` â€” `{{</* asset-image src="..." alt="..." width="..." */>}}` shortcode for embedding `assets/images/` images inside markdown bodies with Hugo image processing. Errors if `src` doesn't resolve.

### Content model

- `content/_index.md` â€” home (title only; body is unused, homepage is layout-driven).
- `content/about-us.md`, `content/contact.md` â€” top-level pages.
- `content/services/_index.md` + `content/services/<slug>.md` â€” service section.
- Markdown uses **unsafe goldmark** (`hugo.yaml`: `markup.goldmark.renderer.unsafe: true`) â€” service pages embed raw `<div class="service-features-grid">` HTML inside markdown. This is intentional; don't strip it.

**Service front matter contract** (load-bearing for the data-driven nav/cards):

```yaml
title: "Yacht"
image: "yacht.jpg"        # filename under assets/images/services/
icon: "M3 13.5L12 3..."   # SVG path "d" attribute, rendered inside <svg viewBox="0 0 24 24">
description: "..."        # SEO meta description
summary: "..."            # marketing copy used on home/listing cards (overrides Hugo's auto .Summary)
weight: 1                 # sort order across nav, dropdown, footer, sidebar, cards
```

`image` is a filename, not a path â€” the template prefixes `images/services/`. `icon` is just the path `d` data, not full SVG markup.

### Site params

`hugo.yaml` `params`:

- Flat: `phone`, `email`, `address`, `description`, `og_image` (filename under `assets/images/`).
- Nested `business:` â€” `legalName`, `streetAddress`, `postalCode`, `locality`, `country` (ISO 3166-1 alpha-2), `areaServed`, `latitude`, `longitude`. Drives the JSON-LD LocalBusiness emitted from `structured-data.html`.

`disableKinds: [taxonomy, term]` is set because the site has no categories/tags and we don't want `/categories/` or `/tags/` polluting the sitemap.

### Styling

- `assets/css/main.css` â€” single hand-written stylesheet, no framework. Brand tokens are CSS custom properties at `:root`: `--navy` `#0a1628`, `--gold` `#c4922a`, fonts `Playfair Display` (serif headings) + `Source Sans 3` (sans body).
- `assets/js/main.js` â€” vanilla JS for navbar scroll state, mobile menu toggle (with `aria-expanded` sync), dropdown toggle.
- Both pass through `resources.Minify | resources.Fingerprint` in `baseof.html` and emit `integrity="sha256-..."` for SRI â€” edit the `assets/` source, not `public/`.

### Accessibility

- `.skip-link` in `baseof.html` jumps focus to `#main` (the `<main>` element) â€” visually hidden until focused.
- `nav-toggle` keeps `aria-expanded` in sync with the `.open` class on the menu (see `main.js`).
- Outstanding: services dropdown is mouse-hover-only on desktop â€” no keyboard support yet.

### Images

- All site images live under `assets/images/` (services in `assets/images/services/`). Hugo Pipes transcodes them to webp + fingerprints the output (cache-busting via filename hash).
- Service templates and the home page card use `.Process "webp q85"` (format-only, no resize) so the four service images and one about-us image dedupe to one built artifact each.
- The `asset-image` shortcode is the only way to surface `assets/images/` content from inside a markdown body â€” direct `<img src="/images/...">` skips the pipeline and breaks if filenames change.
- Source images are currently 600Ă—537 (services) / 960Ă—540 (about-us). Templates do **not** upscale; if you want sharper detail on hi-DPI displays, replace the source with a larger image â€” templates will pick it up automatically.

### Multilingual

Two languages configured in `hugo.yaml`: `en` (default, served at `/`) and `nb` (Norwegian BokmĂĄl, served at `/nb/`). `defaultContentLanguageInSubdir: false` keeps English at the root.

- **UI strings** â†’ `i18n/en.yaml` and `i18n/nb.yaml`. Flat snake_case keys (`nav_home`, `form_email`, etc.). Templates reference via `{{ i18n "key" }}`. Page bodies and marketing copy do NOT belong here â€” those live in `content/*.md`.
- **Content translations** â†’ `*.nb.md` siblings of `*.md` files. Hugo auto-links them as translations of the same logical page. A missing translation falls back to a 404 â€” there's no automatic English fallback for missing pages.
- **Current state** â€” Norwegian stubs exist for top-level pages (home, about-us, contact, services index) with placeholder bodies marked `<!-- TODO -->`. Service detail pages (`yacht.nb.md` etc.) are not yet created; `/nb/services/<slug>/` will 404 and the Norwegian services list/dropdown will be empty until you create them.
- **Language switcher** lives in `partials/header.html` topbar, ranges over `.AllTranslations`. Marks current with `class="active"` + `aria-current="true"`.
- **Per-language params** â€” each language block sets its own `description` (for `<meta description>` and OG description) and `og_locale` (`en_US` / `nb_NO`, Facebook-format territory codes). Other params (`phone`, `email`, `address`, `business.*`) are shared at the top level.
- **Internal links** must go through `relLangURL` (e.g. `{{ "/about-us/" | relLangURL }}`) so they prefix `/nb/` correctly when rendered in Norwegian context.
- **Form JS strings** are passed from Hugo to JS via `data-msg-*` attributes on `#contact-form` so the JS reads the localized text rather than hardcoding English.

### Adding a new UI string

1. Add the key + English text to `i18n/en.yaml`.
2. Add the same key + Norwegian to `i18n/nb.yaml` (mandatory â€” Hugo logs a warning at build if a key is missing in a language).
3. Reference as `{{ i18n "key" }}` in templates.

### Translating a content page

1. Copy `content/foo.md` â†’ `content/foo.nb.md`.
2. Translate the `title`, `description`, and body.
3. Leave `image`, `icon`, `weight` as-is (those are language-independent).

## Adding a new service

One file, one image. Nav/cards/dropdown/sidebar/footer all update automatically.

1. Drop the image into `assets/images/services/<slug>.jpg` (or .png).
2. Create `content/services/<slug>.md` with the front-matter contract above (`title`, `image`, `icon`, `description`, `summary`, `weight`).

The home grid CSS is sized for 4 cards â€” adding a 5th may need a CSS tweak in `assets/css/main.css` `.services-grid`.

## Gotchas

- `static/images/` exists but is empty â€” Hugo Pipes (`assets/`) is the canonical image location now.
- `.hugo_build.lock` is a Hugo runtime artifact; leave it alone.
- `public/` is build output (ignored by git); never edit it by hand.
- `resources/_gen/` is the image-processing cache (also ignored). Safe to delete; Hugo will regenerate.
- Local Hugo install: this machine's snap is broken (`/snap/hugo` missing `meta/snap.yaml`). Use `~/.local/bin/hugo` (Hugo Extended 0.151.0 installed manually). Repair the snap with `sudo snap refresh hugo` when convenient.
